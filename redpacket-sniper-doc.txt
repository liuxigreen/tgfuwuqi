AgentHansa 红包狙击系统 - 逻辑说明
====================================

脚本文件: agenthansa_redpacket_sniper.py (893行, 35KB)
创建时间: 2026-04-12
状态: 稳定运行 ✅

一、核心架构
-----------

1. 守护进程模式
   - PID文件: /root/.hermes/agenthansa/memory/sniper.pid
   - 状态文件: /root/.hermes/agenthansa/memory/sniper-state.json
   - 日志文件: /root/.hermes/agenthansa/memory/sniper.jsonl
   - 学习数据: /root/.hermes/agenthansa/memory/sniper-learn.json

2. 启动方式
   - 后台运行: tmux new-session -d -s sniper "python3 agenthansa_redpacket_sniper.py daemon"
   - 单次运行: python3 agenthansa_redpacket_sniper.py

二、核心逻辑流程
----------------

1. 查询红包 (GET /red-packets)
   - 获取活跃红包列表
   - 获取下次红包时间 (next_packet_seconds)
   - 清理7天前的尝试记录

2. 状态检查
   - 跳过已尝试的红包 (attempted_red_packets)
   - 跳过重试冷却中的红包 (packet_retry_after_epoch)

3. 预热机制
   - ref link预热: ensure_recent_ref_link()
   - alliance提交预热: ensure_recent_alliance_submission()
   - 论坛评论预热: ensure_recent_forum_comment()
   - 论坛投票预热: ensure_recent_forum_vote()

4. 解题逻辑 (优先级)
   - 第1级: 本地规则引擎 (solve_question_local)
     * 数学计算: 2+2, 3*4, sqrt(16)
     * 布尔逻辑: true/false, yes/no
     * 固定答案: AgentHansa相关问题
   
   - 第2级: 廉价模型 (answer_question_with_cheap_model)
     * 使用Haiku或Sonnet
     * 防止LLM浪费: 题目过长或过于复杂时跳过
     * 验证答案格式: 必须是数字/布尔/单词

5. 提交答案 (POST /red-packets/{id}/join)
   - 发送答案: {'answer': ans}
   - 记录成功/失败

6. 失败重试机制
   - ref link错误 → 强制刷新ref link → 重新解题
   - alliance提交错误 → 强制刷新alliance提交 → 重新解题
   - 论坛投票错误 → 强制刷新论坛投票 → 重新解题
   - 答案错误 → 记录失败，下次不再尝试

7. 休眠唤醒
   - 计算下次红包时间
   - 提前60秒唤醒准备
   - 使用time.sleep()精准休眠

三、解题引擎详解
----------------

1. 本地规则引擎 (agenthansa_quiz.py)
   - 数学计算: eval()安全执行
   - 布尔判断: 关键词匹配
   - 固定答案: 硬编码常见问题

2. LLM解题 (agenthansa_llm.py)
   - 优先使用Haiku (便宜)
   - 备选使用Sonnet (质量高)
   - 防止浪费: 题目长度限制, 答案格式验证

四、状态管理
------------

1. attempted_red_packets
   - 记录已尝试的红包ID和时间
   - 7天过期自动清理

2. packet_retry_after_epoch
   - 记录需要重试冷却的红包
   - 1小时过期自动清理

3. pending_failure_notifies
   - 记录需要通知的失败
   - 6小时过期自动清理

4. sniper-learn.json
   - wake_offset: 唤醒提前量 (15-120秒)
   - success_count: 成功计数
   - fail_count: 失败计数
   - avg_delay_seconds: 平均延迟

五、API调用
-----------

1. GET /red-packets
   - 获取活跃红包列表
   - 返回: {active: [...], next_packet_at, next_packet_seconds}

2. GET /red-packets/{id}/challenge
   - 获取红包题目
   - 返回: {question, mathQuestion, challengeQuestion}

3. POST /red-packets/{id}/join
   - 提交答案
   - 发送: {answer: "答案"}
   - 返回: {success, payout_amount, error}

六、错误处理
------------

1. HTTP 429 (Too Many Requests)
   - 记录重试冷却时间
   - 增加下次尝试间隔

2. HTTP 400 (Bad Request)
   - 检查是否需要ref link
   - 检查是否需要alliance提交
   - 检查答案是否错误

3. 解题失败
   - 本地规则失败 → 尝试LLM
   - LLM失败 → 跳过该红包

4. 网络错误
   - 增加错误计数
   - 连续错误超过阈值 → 退出守护进程

七、性能优化
------------

1. 解题优先级
   - 本地规则 (0ms) > LLM (1-3s)
   - 避免不必要的LLM调用

2. 状态缓存
   - 缓存ref link (30分钟有效)
   - 缓存alliance提交 (30分钟有效)
   - 缓存论坛投票 (30分钟有效)

3. 休眠策略
   - 精准计算下次唤醒时间
   - 提前60秒唤醒准备
   - 学习调整唤醒提前量

八、监控与通知
--------------

1. TG通知
   - 启动通知: notify_sniper_event('start')
   - 红包总结: notify_redpacket_summary()
   - 失败通知: notify_failure()

2. 日志记录
   - JSONL格式: sniper.jsonl
   - 记录每次运行结果
   - 记录成功/失败详情
